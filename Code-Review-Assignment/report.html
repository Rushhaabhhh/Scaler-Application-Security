Executive Summary

This report details the findings from a comprehensive secure code review of the Scaler Support Desk web application. The review combined automated scan capabilities of Static Application Security Testing (SAST) tools, specifically Semgrep and Bandit, with exhaustive manual code inspection.

A total of 13 distinct findings were identified. Through manual verification, 10 findings were confirmed as True Positives (exploitable vulnerabilities) representing critical security risks like Remote Code Execution (RCE), OS Command Injection, SQL Injection, Stored Cross-Site Scripting (XSS), and Insecure Direct Object References (IDOR).

Crucially, 3 findings were classified as False Positives. While flagged by automated scanners due to insecure code patterns, manual validation confirmed that robust validation checks, containment filters, and allowlists implemented in the application logic successfully mitigate these concerns.

1. SQL Injection in Ticket Search True Positive
File: supportdesk-app/models.py
Function: search_tickets(term)
Vulnerable Line(s): Line 117-118
Severity: Medium
Screenshot of SAST Tool Output

>> Issue: [B608:hardcoded_sql_expressions] Possible SQL injection vector through string-based query construction. Severity: Medium Confidence: Low CWE: CWE-89 (https://cwe.mitre.org/data/definitions/89.html) Location: supportdesk-app/models.py:117:10 116 conn = get_connection() 117 sql = "SELECT * FROM tickets WHERE subject LIKE '%" + term + "%'" 118 rows = conn.execute(sql).fetchall()
Screenshot of Vulnerable Code
supportdesk-app/models.py python
115def search_tickets(term):
116 conn = get_connection()
117 sql = "SELECT * FROM tickets WHERE subject LIKE '%" + term + "%'"
118 rows = conn.execute(sql).fetchall()
119 conn.close()
120 return rows
Why is it Vulnerable?

The function accepts a user-controlled parameter term and builds an SQL statement by concatenating this string parameter directly into the SQL query string. The query is then executed on the database database connection using conn.execute(sql).

Because there is no sanitization, escaping, or parameterization, any characters with SQL semantic meaning (such as quotes, semicolons, or comment indicators) will be interpreted as SQL commands by the SQLite parser. For example, if a user enters %' UNION SELECT 1,2,password_hash,role,5 FROM users --, the query becomes:
SELECT * FROM tickets WHERE subject LIKE '%%' UNION SELECT 1,2,password_hash,role,5 FROM users --%' This lets the attacker pull user passwords and roles straight out of the database.

Security Impact

Information Disclosure: Attackers can read arbitrary data from any table in the database, including password hashes, credit levels, and other users' private support tickets. Depending on database write privileges, it can also lead to data modification or deletion.

Recommended Remediation

Implement parameterized queries (also known as prepared statements). SQLite will treat the user input strictly as a literal value rather than executable code.

116 conn = get_connection()

Why is it Vulnerable?

The application hashes user passwords using the MD5 algorithm. MD5 is an outdated, fast hashing algorithm designed for speed, not cryptography.

MD5 is highly susceptible to collision attacks (generating different inputs that result in the same hash) and precomputed rainbow tables. In addition, because MD5 is extremely fast, an attacker who obtains the database can run billions of guesses per second on standard hardware, exposing user passwords in seconds. The code also lacks salting, which means identical user passwords result in the same hash, exposing common passwords.

Security Impact

Account Compromise: If database entries are leaked or read via SQL injection, attackers can crack user hashes to recover plaintext passwords and compromise accounts across other platforms (credential stuffing).

Recommended Remediation

Replace MD5 with a secure, slow password hashing algorithm with built-in salting and work factor configuration. Recommended choices include bcrypt, Argon2, or Flask's native generate_password_hash utility.

Why is it Vulnerable?

The application features several state-changing POST forms (e.g. login, comment submission, file upload, and ticket escalation). However, the application templates render these forms without any CSRF tokens, and the backend has no CSRF token validation.

An attacker can build a malicious external site that contains a form auto-submitting to the target server's endpoints (e.g. http://localhost:5050/tickets/1/escalate). If an authenticated victim visits the malicious site, their browser automatically includes the session cookie in the POST request, allowing the attacker to perform actions on their behalf.

Security Impact

Cross-Site Request Forgery (CSRF): Unauthorized actions performed on behalf of authenticated users, such as posting comments, file uploads, and unauthorized ticket escalation.

Recommended Remediation

Integrate CSRF protection using a library like Flask-WTF to enforce CSRF validation globally. In HTML forms, render the csrf token:

Remediated supportdesk-app/templates/ticket.html html
14 <form method="post" action="/tickets/{{ ticket.id }}/escalate">
14.1 <input type="hidden" name="csrf_token" value="{{ csrf_token() }}">
